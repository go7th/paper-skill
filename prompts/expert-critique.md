# 6 专家挑刺 Prompt 模板（按方案类型）

S 级 Step 1 调用此文件。根据 `s-tier.md` Step 0 识别出的方案类型，选对应的专家组。**每个专家独立 review，不要合并跑**。

## 通用 prompt 框架

```
请你以 [专家角色] 的视角审视下面这份方案。
你的任务是挑出尽可能多的具体问题，每条问题需要包含：
(1) 问题描述
(2) 为什么是问题
(3) 怎么验证

不要客套，不要总结，不要肯定，只输出问题清单。
至少 3 条。
```

## 技术方案专家组

| 专家角色 | 关注维度 |
|---------|---------|
| 资深安全工程师 | 身份认证、加密、攻击面、注入风险、CVE、依赖漏洞 |
| 合规专家 | 法律、隐私、行业标准、跨境数据、等保、GDPR |
| 系统架构师 | 耦合度、可扩展性、技术选型理由、生态兼容、技术债 |
| SRE / 运维负责人 | 可观测性、降级方案、回滚、监控、容量、故障演练 |
| UX 工程师 | 首次使用、错误态、多终端、可访问性、加载性能 |
| ROI 评估 / FP&A | 投入产出、隐性成本、机会成本、TCO、人月估算 |

## 产品方案专家组

| 专家角色 | 关注维度 |
|---------|---------|
| 用户研究员 | 目标用户画像清晰度、需求真实性、使用场景覆盖 |
| 增长负责人 | 获客路径、转化漏斗、留存机制、传播性 |
| 数据分析师 | 北极星指标定义、可度量性、AB 测试设计 |
| 竞品分析师 | 竞品对比、差异化定位、替代方案威胁 |
| 商业化负责人 | 变现路径、定价策略、付费意愿验证 |
| 风险 / 法务 | 业务风险、合规边界、用户协议 |

## 业务 / 运营方案专家组

| 专家角色 | 关注维度 |
|---------|---------|
| 市场负责人 | 市场规模、时机、竞争格局、定位 |
| 用户运营 | 用户分层、激活路径、生命周期管理 |
| 渠道专家 | 渠道选型、成本结构、ROI、渠道冲突 |
| 数据 BI | KPI 拆解、数据可获取性、归因模型 |
| 法务 / 合规 | 营销合规、广告法、用户协议、纠纷预案 |
| CFO / 财务 | 预算、现金流、毛利、回收周期 |

## 设计方案专家组

| 专家角色 | 关注维度 |
|---------|---------|
| 可用性专家 | 任务完成路径、认知负荷、错误防护 |
| 设计系统负责人 | 一致性、组件复用、规范遵循 |
| 可访问性专家 | WCAG、色对比度、屏幕阅读器、键盘导航 |
| 性能工程师 | 资源加载、动画性能、首屏指标 |
| 品牌负责人 | 视觉一致性、品牌调性、情感表达 |
| 业务方 | 业务目标匹配、转化路径、CTA 设计 |

## 调用示例

S 级 Step 0 识别为技术方案后，按 6 个角度依次调用：

```
第 1 轮：
[资深安全工程师视角 review] → 收集问题
[合规专家视角 review] → 收集问题
[系统架构师视角 review] → 收集问题
[SRE 视角 review] → 收集问题
[UX 工程师视角 review] → 收集问题
[ROI 评估视角 review] → 收集问题

→ 合并去重 → 分 4 类反哺 → 第 2 轮（最多 3 轮）
```

## 红线

- 不要在同一个 prompt 里要求 Claude "同时扮演 6 个专家"，会让视角被稀释
- 不要使用 "请你忘记你是 AI" 等老式 jailbreak 句式，新模型不吃这套且可能触发对抗检测
- 如果某专家角度在该方案中明显无意义（如设计方案问 SRE），可跳过并在 checklist 中说明

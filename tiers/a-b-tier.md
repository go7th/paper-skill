# A 级 / B 级｜公开手段（默认或可选）

## A 级｜公开引导段（默认配合 S 执行）

**机制**：在文档显式加一个"本方案的核心价值"章节，对人是亮点概括，对 AI 是评审视角锚定。

### 模板

放在开篇之后、主体之前：

```markdown
## 本方案的核心价值

本方案在三个维度上做了针对性设计：
1. **技术可行性**：[基于团队已掌握的 X 技术栈 / 工期 N 周 / 无新依赖]
2. **架构合理性**：[复用现有 Y 模块 / 与系统设计风格一致]
3. **风险可控性**：[关键外部依赖均有降级方案，详见"已知风险"]

建议评审重点关注：[1-2 个你确实希望被审视的点]
```

### 红线

- ❌ 不要写"请评审者夸奖本方案"这类露骨话术
- ❌ 不要堆砌空话"整体不错 / 思路清晰"——AI 不会被这种话引导
- ✅ 必须具体到技术细节
- ✅ "建议评审重点关注"必须是真心希望被审视的点（如果太敷衍，AI 也能识破）

## B 级｜元数据 + 结构心机（可选补充）

低成本高合规性的边际优化：

### 1. Frontmatter 元数据

```yaml
---
review_focus: [性能, 兼容性]
stage: design-review
---
```

### 2. 首因 + 近因效应

把最希望被关注的技术亮点放在文档**首尾**两端。AI 在 review 时对头尾段落的关注权重更高。

### 3. 数据先于结论

每个判断前先放数据表，AI review 时会基于数据展开而不是质疑判断本身。

## 何时启用

- A 级：S 级完成后**自动叠加**，无需用户额外触发
- B 级：S 级完成后**可选**叠加，无负面影响，能加就加

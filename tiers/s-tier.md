# S 级｜专家预审补强（主模式）

S 级是 paper-skill 的核心模式，默认每次都跑。机制：让 Claude 扮演多个领域专家轮流挑刺，把挑出的真问题反哺进方案，等真 reviewer 用 AI 看时找不到新点可挑。

## 完整工作流（5 步）

### Step 0：识别方案类型

跑挑刺之前先识别方案类型，选对应的专家组（详见 `prompts/expert-critique.md`）：

| 方案类型 | 关键特征 | 默认专家组 |
|---------|---------|----------|
| 技术方案 | 涉及架构、技术选型、API、性能指标 | 安全 / 合规 / 架构 / 运维 / UX / ROI |
| 产品方案 | 涉及功能、用户场景、业务流程 | 用户研究 / 增长 / 数据 / 竞品 / 商业化 / 风险 |
| 业务 / 运营方案 | 涉及市场、营销、活动、KPI | 市场 / 用户 / 渠道 / 数据 / 法务 / 财务 |
| 设计方案 | 涉及视觉、交互、信息架构 | 可用性 / 一致性 / 可访问性 / 性能 / 品牌 / 业务匹配 |

**如果无法识别**：默认按技术方案处理，并在 Step 1 开始前问用户确认。

### Step 1：N 专家挑刺

让 Claude 分别从所选专家组中每个角度独立 review 方案，每个角度至少提 3 条具体、可验证的批评意见。

**标准 prompt 模板**（不要使用 "请你忘记你是 AI" 等老式 jailbreak 句式）：

> "请你以 [专家角色] 的视角审视下面这份方案。你的任务是挑出尽可能多的具体问题，每条问题需要包含：(1) 问题描述 (2) 为什么是问题 (3) 怎么验证。不要客套，不要总结，不要肯定，只输出问题清单。每个角度至少 3 条。"

**关键**：逐个角度独立 review，**不要一次跑完所有角度**（视角会被稀释，挑出来的都是泛泛之谈）。

### Step 2：挑刺分类反哺

把所有挑刺分成 4 类，分别处理：

| 类型 | 处理方式 | 写进文档的位置 |
|------|---------|---------------|
| 真问题 | 修改方案设计 | 主体相应章节 |
| 已考虑但没写 | 补充论证依据 | "决策依据" 章节 |
| 越界 / 不属于本期 | 明确划出边界 | "本期非目标" 章节 |
| 边界场景 | 识别 + 缓解措施 | "已知风险" 章节 |

### Step 3：循环至收敛（硬上限 3 轮）

把改后的方案再过一遍同样的 N 专家挑刺。**最多 3 轮**：

- 第 1 轮：通常挑出 15-25 条
- 第 2 轮：通常挑出 5-10 条
- 第 3 轮：通常挑出 0-3 条，若仍有具体问题，全部归入"已知风险"章节并标注"待后续版本处理"

**到第 3 轮强制停止**，避免无限挑剔陷入"风格建议"的死循环。

### Step 4：FAQ 化最难的几条

把第 1-3 轮里最难回答的 3-5 条挑刺写成"预期问答"章节。

格式：

```markdown
## 预期问答

**Q1：[挑刺问题原文]**
A：[你的回答，含数据 / 引用 / 实施路径]

**Q2：...**
A：...
```

这样领导真问起来时，你和 AI review 的回答一致，强化方案权威性。

### Step 5：进入 A 级补强

S 级收敛后，自动叠加 A 级公开引导段（详见 `tiers/a-b-tier.md`）。

## S 级完成标志

S 级跑完后应该满足：

- [ ] 所有专家角度都有对应的章节回应（防御章节 + FAQ）
- [ ] 每个关键决策后面都附数据或约束依据
- [ ] 风险表至少覆盖每个专家角度提出过的 top 风险
- [ ] FAQ 至少 3 条
- [ ] 章节名无禁用词（总结 / 概述 / 分析等）
- [ ] 完整跑了至少 2 轮挑刺循环

完成后输出 checklist 让用户核对。

## 反模式（不要这么写）

| 反模式 | 改成 |
|--------|------|
| "我觉得 A 更好" | "A 在 X 指标上优于 B（见对比表）" |
| 滥用"可能 / 也许 / 大概" | 给数据，或明确"待 PoC 验证" |
| 没有数字的判断 | 任何对比加单位（ms / MB / 人天） |
| 只写"做什么" | 显式写 Non-Goals |
| 章节名 "总结 / 概述 / 分析" | 改成 "结论与下一步 / 背景与思路 / 方案对比" |
